## 1. 产品概述

广骏供应链服务是一个面向中美跨境集运业务的 Logistic OS。系统主要服务在美国使用微信的华人用户，帮助用户在中国电商平台下单后，将商品寄到国内仓，再由商家完成入库检查、拍照确认、计重计体积、支付状态确认、国际发货和物流追踪。

系统由三个主要应用组成：

1. `backend`：NestJS API 服务，负责业务逻辑、数据权限、微信登录、订单流转、图片上传、通知、发货等。
2. `frontend`：Next.js 外部网站和未来推荐系统，面向美国本地华人，有 SEO 需求，也可承载 Admin 后台。
3. `miniprogram`：微信小程序，面向下单和查询物流的客户，用于查看地址、订单、包裹入库照片、确认包裹、提交异常和查看物流状态。

## 2. 产品定位

### 2.1 一句话定位

一个为中美跨境集运业务设计的可视化物流履约系统。

### 2.2 核心价值

1. 建立信任：通过入库拍照、面单照片、货物照片和用户确认机制，让客户看得见每一步。
2. 提高效率：通过标准化入库、确认、支付、发货和异常处理流程，减少人工沟通成本。
3. 提供性价比：通过计重、计体积、合单和透明费用展示，帮助客户理解费用构成。

### 2.3 品牌策略

- 小程序注册/审核名称优先使用：广骏供应链服务。
- 面向用户的品牌展示可使用：广骏国际快运。
- 宣传语使用：看得见的跨境物流。
- 避免使用高风险文案：代购、免税、避税、灰关、走私、低价清关等。
- 推荐页面文案：提供跨境供应链与物流信息服务；提供仓储入库、包裹确认、发货管理与物流查询服务。

## 3. 产品目标

### 3.1 用户侧目标

用户可以：

- 通过微信小程序完成微信登录。
- 获取并复制国内仓收货地址。
- 在淘宝、京东等平台下单时使用仓库地址。
- 查看国内仓是否收到货。

- 查看包裹重量、尺寸、体积重、计费重量。
- 对包裹进行确认或提交异常。
- 查看支付状态。
- 查看国际发货渠道、运单号和物流状态。
- 在最终交付时通过二维码进行本人确认。

### 3.2 管理员侧目标

管理员可以：

- 创建和管理用户订单。
- 录入国内包裹入库信息。
- 上传包裹照片。
- 填写重量和长宽高。
- 管理订单和包裹状态。
- 手动确认支付状态。
- 在必要时使用 override 后门强制推进状态。
- 创建国际发货信息。
- 记录出货时间、物流渠道、运单号。
- 处理异常件。
- 查看操作日志，追溯关键操作。

### 3.3 平台侧目标

系统需要：

- 保持清晰的数据模型。
- 保证用户只能访问自己的订单和包裹。
- 保证所有管理员关键操作可追踪。
- 保留未来接入第三方物流 API 的能力。
- 支持未来数据看板和运营分析。
- 支持未来拓展多仓库、多渠道、多价格规则。

## 7. 核心状态机

### 7.1 Order Status

```text
UNINBOUND
→ INBOUNDED
→ USER_CONFIRM_PENDING
→ REVIEW_PENDING
→ PAYMENT_PENDING
→ PAID
→ READY_TO_SHIP
→ SHIPPED
→ COMPLETED
```

说明：

| 状态 | 中文 | 含义 |
|---|---|---|
| UNINBOUND | 未入库 | 订单创建后，包裹尚未到国内仓 |
| INBOUNDED | 已入库 | 国内仓已收到至少一个包裹 |
| USER_CONFIRM_PENDING | 待用户确认 | 已上传照片，等待用户确认 |
| REVIEW_PENDING | 待审核 | 用户确认后等待管理员审核 |
| PAYMENT_PENDING | 待支付 | 费用已计算，等待用户完成外部支付 |
| PAID | 已支付 | 管理员确认支付完成 |
| READY_TO_SHIP | 待发货 | 可以安排国际发货 |
| SHIPPED | 已发货 | 已创建国际物流记录 |
| COMPLETED | 已完成 | 用户确认收货或系统完成 |

### 7.2 Package Status

```text
CREATED
→ INBOUNDED
→ USER_CONFIRM_PENDING
→ CONFIRMED
→ EXCEPTION
→ SHIPPED
```

### 7.3 Payment Status

```text
UNPAID
PROCESSING
PAID
```

业务规则：

- `UNPAID`：不安排发货。
- `PROCESSING`：不安排发货。
- `PAID`：允许进入待发货或发货流程。
- 管理员 override 可绕过规则，但必须记录日志和原因。

## 8. 端到端业务流程

### 8.1 用户下单到国内仓

1. 用户进入小程序。
2. 用户微信登录。
3. 系统生成四位 user_code。
4. 用户进入地址页。
5. 用户复制国内仓地址。
6. 地址中应包含用户识别信息，例如收件人字段中包含 user_code。
7. 用户在淘宝、京东等平台下单。
8. 商品发往国内仓。

### 8.2 国内仓入库

1. 管理员收到包裹。
2. 管理员用后台根据 user_code、国内快递单号或其他信息匹配用户和订单。
3. 管理员录入包裹信息。
4. 管理员上传照片：外包装、面单、内部物品。
5. 管理员录入重量和尺寸。
6. 系统计算体积重。
7. 系统更新状态为待用户确认。
8. 系统可触发微信订阅消息提醒用户。

### 8.3 用户确认

1. 用户进入订单详情页。
2. 用户查看包裹照片、重量和尺寸。
3. 用户点击确认无误，包裹状态变为 confirmed。
4. 用户点击有问题，系统创建 ExceptionCase，并暂停后续流程。
5. 若用户 48 小时未确认，可由系统自动确认，或标记为待管理员处理。MVP 默认先做手动配置，不强依赖定时任务。

### 8.4 审核与支付

1. 管理员确认包裹信息。
2. 系统计算费用。
3. 订单进入待支付。
4. 用户通过外部渠道完成支付。
5. 管理员后台标记支付状态为已支付。
6. 订单进入待发货。

### 8.5 国际发货

1. 管理员选择物流渠道：UPS、DHL、EMS、海运、空运或其他。
2. 管理员录入运单号。
3. 管理员填写出货时间和预计到达时间。
4. 系统更新订单为已发货。
5. 用户可在小程序查看物流渠道和运单号。

### 8.6 收货确认二维码

1. 系统为订单生成二维码。
2. 二维码绑定 order_id、token 和下单用户 openid。
3. 用户使用原微信账号扫描二维码。
4. 后端校验扫码用户是否为原订单用户。
5. 是本人：允许确认收货或更新状态。
6. 非本人：返回提示，不更新任何状态。

## 10. 运费计算需求

### 10.1 体积重公式

```text
volume_weight = length_cm * width_cm * height_cm / 6000
```

### 10.2 计费重量

```text
chargeable_weight = max(actual_weight, volume_weight)
```

### 10.3 费用

```text
price = unit_price * chargeable_weight
```

MVP 可以由管理员手动设置 unit_price 或 final_price。后续可做价格规则引擎。

## 12. 安全与权限需求

### 12.1 小程序用户权限

- 用户只能访问自己的数据。
- 后端通过 JWT 识别 user_id。
- openid 只在后端通过微信 code2session 获得。
- 前端不能伪造 openid。

### 12.2 管理员权限

- 管理员登录独立于微信用户。
- MVP 可先做单一 admin role。
- 后续支持 SUPER_ADMIN、OPERATOR、SUPPORT 等角色。

### 12.3 二维码权限

- QR token 存储 hash。
- 扫码时通过当前用户 openid 校验。
- 非本人扫码不更新状态。
- 扫码日志必须记录。

### 12.4 图片权限

- 上传由后端生成授权或由后端代传。
- 图片 metadata 必须存数据库。
- 用户只能查看属于自己订单的图片。

## 16. 验收标准

MVP 通过验收需要满足：

1. 用户可以用微信登录并获得 user_code。
2. 用户可以复制仓库地址。
3. 管理员可以创建订单和包裹。
4. 管理员可以上传至少三类图片。
5. 用户可以查看自己的包裹照片。
6. 用户可以确认包裹或提交异常。
7. 管理员可以标记支付状态。
8. 未支付订单默认不能发货。
9. 管理员 override 可以强制发货，并记录日志。
10. 用户可以查看发货渠道和运单号。
11. 非本人不能查看他人订单。
12. 后端 API 可部署到 AWS EC2。
13. 数据库使用 Supabase Postgres。
14. 图片使用 Supabase Storage。
15. 外部站点使用 Next.js 并可部署到 Vercel。
